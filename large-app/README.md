# Chicken-and-egg problems with CloudFormation not reflecting parent/child relationships

This single flat stack of 140+ resources includes a Lambda layer (`UtilsLayer`) whose content is referenced as a ZIP artifact already present in an S3 bucket (`cfn-deploy-{AccountId}-{Region}`). That bucket and its upload are prerequisites for the stack to deploy, but they are not CloudFormation resources — they are an external setup step that must happen first. Discovery sees the layer resource and the bucket name in its properties, but has no visibility into the script or pipeline that performed the upload, nor into the dependency ordering that made it possible. A migration that imports only what discovery returns will reproduce the layer declaration without the pre-deployment artifact stage, resulting in code that cannot deploy on a fresh account without first recreating the invisible prerequisite.
