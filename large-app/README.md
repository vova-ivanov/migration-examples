# Chicken-and-egg problems with CloudFormation not reflecting parent/child relationships

This single flat stack of 140+ resources includes a Lambda layer (`UtilsLayer`) whose content is referenced as a ZIP artifact already present in an S3 bucket (`cfn-deploy-{AccountId}-{Region}`). That bucket and its upload are prerequisites for the stack to deploy, but they are not CloudFormation resources — they are an external setup step that must happen first. Discovery sees the layer resource and the bucket name in its properties, but has no visibility into the script or pipeline that performed the upload, nor into the dependency ordering that made it possible. A migration that imports only what discovery returns will reproduce the layer declaration without the pre-deployment artifact stage, resulting in code that cannot deploy on a fresh account without first recreating the invisible prerequisite.

## What this app simulates

`large-app` models a simplified online store backend. Users register and authenticate (via Cognito-backed sessions), browse and manage items (a product catalog with owner and status indexes), and place orders (linked to users via a GSI). Product images and attachments are uploaded directly to S3 via presigned URLs generated by the API.

Order fulfillment runs asynchronously: the API enqueues work to SQS, a Worker Lambda processes jobs in batches of 10, and a Notifier Lambda sends confirmation emails via SES. A nightly Step Functions state machine (validate → process → notify) handles batch reconciliation, while a separate cleanup job purges expired sessions and stale orders on a daily schedule.

Payment processing is intentionally out of scope — the stack models the catalog, checkout flow, and fulfillment pipeline without a payment provider, making it a self-contained example of the serverless store pattern.
