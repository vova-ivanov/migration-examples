# CloudFormation nested stack hierarchy being flattened on import

This application is deliberately structured as a hierarchy of nested stacks — network, storage, auth, compute, messaging, and monitoring each live in a child stack with explicit cross-stack output references. That layering is an intentional architectural boundary: it enforces deployment order, scopes IAM, and makes the dependency graph explicit. Stack discovery has a tendency to flatten this hierarchy and present all child resources as if they belong to a single top-level stack, discarding the nesting structure. The result is a migration that loses the organizational intent and the cross-stack output wiring, requiring the engineer to manually reconstruct the module boundaries that were already clearly expressed in the original template hierarchy.
